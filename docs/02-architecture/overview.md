## 二、三端职责对比

| 维度 | C 端(用户) | 代养人端 | 管理后台 |
|---|---|---|---|
| **平台** | 小程序+APP+H5 | 同 C 端(角色切换) | 桌面 Web |
| **技术** | uni-app | 同 C 端 | Vue3 + Vite |
| **主要场景** | 认领 / 看猪 / 拼猪 / 收货 | 喂养打卡 / 健康上报 | 运营 / 客服 / 财务 |
| **登录方式** | 微信授权 | 微信授权 + 农户实名 | 用户名密码 + 2FA |
| **设备** | 手机 | 手机(老年友好) | 电脑 |
| **预期使用频次** | 一天 1-2 次(看猪) | 一天 3 次(三餐打卡) | 工作日全天 |

## 三、核心数据流(以「认一头猪 + 拼猪」为例)

```
1. [用户] 小程序首页 → 点「立即认养」
                              │
                              ▼
2. [用户] 选品种 → 选农户 → 选保险 → 确认订单
                              │
                              ▼
3. [前端] POST /api/orders { breed, farmer_id, insurance }
                              │
                              ▼
4. [Auth] 校验 JWT → 提取 userId
                              │
                              ▼
5. [OrderService]
   - 创建 order(status=pending_payment)
   - 调微信支付下单
                              │
                              ▼
6. [Wx Pay] 用户支付成功
                              │
                              ▼
7. [Pay Webhook] /api/pay/notify
   - 更新 order.status = active
   - 给 user 钱包预扣额度
   - 从 available 池随机分配一只 pig
   - 创建 pig 与 order 的绑定
                              │
                              ▼
8. [用户] 想拼猪 → POST /api/orders/{id}/share
                              │
                              ▼
9. [ShareService]
   - 生成 share_code(6位短码)
   - 创建 share record(host=user)
   - 返回 invite URL
                              │
                              ▼
10. [用户] 把 URL 发给亲友 → 亲友点开 → 微信授权
                              │
                              ▼
11. [前端] POST /api/share/join { code }
                              │
                              ▼
12. [ShareService]
    - 校验 share_code 有效 + 未满员
    - 创建 share_member(user_id, share_id, status=joined)
                              │
                              ▼
13. 双方都能进入「猪只详情页」,看到:
    - 实时直播
    - 喂养记录
    - 健康档案
    - 拼猪成员头像列表
```

## 四、后端模块的依赖关系

```
                            ┌─────────┐
                            │  Auth   │ ◄─── 所有模块依赖(JWT 守卫)
                            └────┬────┘
                                 │
            ┌────────────────────┼────────────────────┐
            ▼                    ▼                    ▼
       ┌────────┐          ┌────────┐           ┌────────┐
       │  User  │          │ Farmer │           │  Wallet│
       └────┬───┘          └────┬───┘           └────┬───┘
            │                   │                    │
            └────────┬──────────┴────────┬───────────┘
                     ▼                   ▼
                ┌────────┐           ┌────────┐
                │  Pig   │ ◄─────────┤  Order │ ──┐
                └───┬────┘           └───┬────┘   │
                    │                    │        │
        ┌───────────┼────────────┐       │   ┌────▼────┐
        ▼           ▼            ▼       │   │  Share  │
   ┌────────┐ ┌─────────┐  ┌─────────┐   │   └────┬────┘
   │ Live   │ │ Feeding │  │ Health  │   │        │
   └────────┘ └─────────┘  └─────────┘   │        ▼
                                         │   ┌─────────┐
                                         └─► │ Message │
                                             └─────────┘
```

依赖原则:
- **Auth** 是底座,所有模块都用它的 `@AuthGuard()` + `@CurrentUser()`
- **Wallet** 只被 Order / Share 调用(扣款 / 退款)
- **Pig** 是核心实体,被 Order / Live / Feeding / Health 引用
- **Message** 是终点(任何模块产生通知都进入这里)

详细模块清单 → [`../03-backend/modules.md`](../03-backend/modules.md)

## 六、规模扩展规划(2027+)

| 阶段 | 触发条件 | 动作 |
|---|---|---|
| **单机** | 当前 | 一台轻量应用服务器 |
| **拆服务** | DAU > 5k | API / 直播流 / 文件 分离到独立机器 |
| **拆库** | 数据 > 50GB | 主从 + 读写分离 |
| **拆地域** | 用户跨大区 | 多地域部署 + CDN |
| **微服务化** | 团队 > 15 人 | 按业务线拆 NestJS 微服务 |

> 现阶段**强烈反对**任何提前微服务化的提议——会让 2 人小团队的开发效率掉一半。
