## 二、后端:为什么选 NestJS

### 选 NestJS 的理由

1. **强结构,反"屎山"** — 控制器 / 服务 / 模块强制分层,小团队一致性最好
2. **TypeScript 原生** — 全栈 TS,前后端类型可共享
3. **依赖注入** — 单元测试与 mock 极其友好
4. **生态完整** — 微信登录 / JWT / WebSocket / 调度 / 队列,都有官方或社区包
5. **NestJS CLI** — `nest g resource user` 一键生成 CRUD 骨架
6. **AI 友好** — Cursor / Claude 对 NestJS 模式识别率最高,我们这个项目高度依赖 AI 辅助

### 替代方案为什么不选

| 候选 | 否决理由 |
|---|---|
| **Express 裸写** | 没结构,2 人 3 个月后代码就乱 |
| **Koa** | 同上 |
| **Egg.js** | 国内,生态卡在 2020,Node 18+ 支持差 |
| **Fastify** | 性能更好,但生态薄,小团队踩坑成本高 |
| **Go + Gin** | 性能怪兽,但前后端语言不一致,招人难,与 AI 协作弱 |
| **Java + Spring Boot** | 太重,启动慢,小项目过度工程 |
| **Python + FastAPI** | 微信生态文档偏 Node,Python 在小程序领域社区弱 |

## 三、客户端:为什么选 uni-app

### 选 uni-app 的理由(决定性)

1. **同一份代码 → 微信小程序 + 安卓 APP + iOS APP + H5 + 其他小程序**
2. 小程序和 APP 业务逻辑 95% 相同,**绝不可能去维护两份**
3. Vue3 + TS,前端工程师上手门槛低
4. DCloud(uni-app 厂商)持续维护,生态稳定
5. 真机调试 / 打包 / 上架,有 HBuilderX 一站搞定

### 替代方案为什么不选

| 候选 | 否决理由 |
|---|---|
| **原生小程序 + 原生 APP** | 两套代码,2 人维护不动,代码必然分叉 |
| **Taro** | 也是跨端方案,但社区文档不如 uni-app,中文资源少 |
| **React Native + 小程序原生** | 仍然需要写两套(RN 不能编译小程序) |
| **Flutter + 小程序原生** | 同上,而且 Flutter 在小程序场景缺位 |
| **mpvue / wepy** | 已停止维护 |
| **Hippy** | 字节系,生态外人难用 |

### uni-app 唯一的代价

- 平台差异处理需要 `#ifdef` 条件编译(每端 5% 差异)
- 某些原生能力需要 uni 原生插件(可控)
- 性能比纯原生差 10–20%(对本项目不敏感)

## 四、数据库:MySQL + Redis

### 主存储为什么是 MySQL

1. 服务器已经装了
2. 业务是**强事务**的(下单 / 支付 / 拼猪 / 钱包),NoSQL 不合适
3. 关系模型清晰(用户—订单—猪—农户—拼猪关系)
4. 团队最熟悉
5. 备份 / 迁移 / 监控生态最成熟

### Redis 用途

| 场景 | 数据结构 | TTL |
|---|---|---|
| 微信登录态缓存 | Hash | 7 天 |
| 短信验证码 | String | 5 分钟 |
| 分享码 | String → orderId | 30 天 |
| 接口限流 | Token Bucket(Lua) | — |
| 直播在线人数 | HyperLogLog / SortedSet | 实时 |
| 任务队列(后续) | List + BullMQ | — |

### 不选其他的理由

| 候选 | 否决理由 |
|---|---|
| **PostgreSQL** | 没硬需求(JSON 索引、地理),团队不如 MySQL 熟 |
| **MongoDB** | 无强事务,业务不合适 |
| **TiDB / 分布式 DB** | 严重过度设计 |

## 五、日志:为什么是 Pino

- **零依赖、高性能**(NestJS 内置 logger 也基于 Pino)
- **JSON 输出**,便于结构化检索(filebeat / loki)
- 开发期 `pino-pretty` 彩色可读

---

## 六、ORM:为什么是 TypeORM(而不是 Prisma)

| 项 | TypeORM | Prisma |
|---|---|---|
| NestJS 集成 | ✅ 官方 | ✅ 社区 |
| TS 类型 | 装饰器写法 | 极强(自动生成) |
| 迁移 | 内置 | 内置(自动 diff) |
| 大表性能 | 中等 | 中等 |
| 国内文档 | 多 | 少 |
| AI 上手 | 熟 | 熟 |

**选 TypeORM 因为**:
- 装饰器写法更"NestJS 感"
- 国内案例多,踩坑有人填过
- 迁移工作流可控

后续如果有迁移意愿,Prisma 是合理替代。

## 七、文件存储

### 短期:服务器本地

- `uploads/` 目录,通过 nginx 静态托管
- 适合 < 1000 张图、< 50 个视频

### 中期:腾讯云 COS

- 域名同源 `cos.rockingwei.online`
- 用 CDN 加速
- 触发上限:头像 / 喂养图片 > 10 万张

### 远期:CDN + COS 分层

## 九、版本锁定原则

- **Node 20.x** —— 服务器 / 本地 / CI 必须一致(用 `.nvmrc`)
- **MySQL 8** —— 不向下兼容 5.7
- **Redis 7** —— 用了 7.0+ 的特性(SET 的 EXAT)
- 每年**统一升级一次**主要依赖,不在迭代中临时升

---

## 十、添加新依赖的纪律

PR 审核里,引入新 npm 包时必须答 4 个问题:

1. **能不能用现有依赖实现?**(防臃肿)
2. **这个包还在维护吗?**(看 weekly downloads / last published)
3. **TS 类型完整吗?**
4. **License 是 MIT/Apache 2.0 吗?**(不能引入 GPL)

不通过,PR 被打回。
